BLE Multi-Role 정책 분석

CW_BLE_MODULE  |  nRF54L15  |  nRF Connect SDK / Zephyr  |  갱신: 2026-03-23

1 전체 아키텍처 — 사내 표준 BLE 모듈

CW_BLE_MODULE은 코웨이 제품(비데, 리모컨, 스마트매트리스, 안마의자 등)에 범용으로 탑재되는 사내 표준 BLE 통신 모듈로 개발 될 예정이다. 제품 개발자는 BLE Stack이나 BLE Service/Profile을 직접 다루지 않으며, HOST MCU ↔ BLE 모듈 간 UART AT 커맨드만으로 BLE 기능을 사용한다.

BLE 역할은 Central + Peripheral 동시 수행(Multi-Role)이 기본이며, AT+ROLE 커맨드로 Peripheral 전용 또는 Central 전용으로 전환 가능하다. 펌웨어는 제품 무관 단일 바이너리(ONE-firmware)로 관리되며, AT+PRODUCTCODE로 전달받은 제품 코드에 따라 런타임에 동작(저장 용량 배분, 데이터 정책 등)이 분기된다.

모든 제품에 동일한 펌웨어 바이너리를 탑재한다. 제품별 차이는 빌드타임이 아닌 런타임에 제품 코드 기반으로 분기한다.

8 다중 연결 슬롯 아키텍처

ONE-firmware 범용 모듈 원칙: CW_BLE_MODULE은 비데, 리모컨, 매트리스, 안마의자 등 모든 제품에 동일한 바이너리로 탑재됩니다. 제품 개발자는 AT 커맨드(AT+ROLE, AT+MLINK)로 역할과 연결 수만 설정하며, Central/Peripheral의 내부 구분을 알 필요가 없습니다.

10 데이터 라우팅 정책 (MULTIDROP / NODEDROP)

레거시 BoT-nLE523 Multicon role의 데이터 통신 정책을 계승합니다. HOST MCU가 AT 커맨드로 데이터 라우팅을 제어하며, BLE 모듈은 인덱스에 따라 해당 장치에 데이터를 전달합니다.

송신 정책

모드AT 커맨드동작
MULTIDROP AT+MDATA=00,<data> 연결된 모든 장치에 동일 데이터 전송 (Central + Peripheral 무관)
NODEDROP AT+MDATA=03,<data> 인덱스 03번 장치에만 데이터 전송
Bypass AT+MBYPASS=ON UART 수신 데이터를 모든 장치에 자동 전달 (AT 구조 없이)

수신 정책

/* 상대 장치에서 데이터 수신 시 HOST MCU에 알림 */
+DATA03[Hello World]     ← 인덱스 03번 장치에서 수신

/* HOST MCU는 인덱스로 어떤 장치인지 식별 가능 */
/* Central/Peripheral 구분 없이 통합 인덱스 사용 */

내부 송신 분기 (모듈 내부, 개발자 비노출)

/* AT+MDATA=idx,data 수신 시 내부 처리 */
struct conn_slot *slot = find_slot_by_idx(idx);

if (slot->is_central) {
    /* Central → 상대 Peripheral: NUS Client Write */
    data_svc_client_send(&slot->client, data, len);
} else {
    /* Peripheral → 상대 Central: NUS Server Notification */
    data_svc_server_send(slot->conn, data, len);
}

제품 개발자는 AT+MDATA=03,data만 사용하면 되며, 03번이 Central인지 Peripheral인지 알 필요가 없습니다.

/* Chunk Header — 8 bytes */
struct chunk_header {
    uint16_t magic;          /* 0xCW01 */
    uint8_t  type;           /* 10=A1010, 20=A1020, 30=A1030 */
    uint8_t  version;        /* 1 */
    uint16_t record_count;
    uint16_t reserved;
};

/* Record — 가변 길이 */
struct chunk_record {
    uint32_t timestamp;      /* UNIX timestamp */
    uint16_t length;         /* payload 길이 */
    uint8_t  payload[];      /* protobuf 직렬화 데이터 */
};

16-4. 경합 지점 분석

공유 자원 접근 스레드 보호 메커니즘 비고

16-5. 개발 시 주의사항

# 규칙 설명
1 파일 I/O는 at_cmd_thread에서만 chunk_mgr, data_mgr API는 현재 단일 스레드 접근을 전제로 뮤텍스가 없다. 다른 스레드에서 호출해야 할 경우 반드시 뮤텍스를 추가하거나, K_MSGQ로 at_cmd_thread에 작업을 위임할 것.
2 UART TX는 반드시 uart_ctrl_send() 경유 s_tx_mutex가 직렬화를 보장한다. uart_poll_out()을 직접 호출하면 출력이 뒤섞일 수 있다.
3 BLE 콜백에서 blocking 금지 BT RX thread는 cooperative(-8)이므로 blocking 호출 시 전체 BLE 스택이 멈춘다. 긴 작업은 k_work_submit()으로 System Workqueue에 위임하거나, K_MSGQ로 다른 스레드에 전달할 것.
4 새 스레드 추가 시 스택 기준 준수 파일 I/O 없음: 768~1024B. 파일 I/O 있음: 2048B 이상. 상세 가이드: docs/스택_깊이_관리_가이드.md 참조.
5 K_MSGQ 큐 오버플로 정책 확인 현재 relay 큐는 K_NO_WAIT + 드롭 정책이다. 큐 크기(32)를 변경하거나 새 큐를 추가할 때 RAM 영향(패킷크기 × 슬롯수)을 §13 RAM 분석 기준으로 검증할 것.
